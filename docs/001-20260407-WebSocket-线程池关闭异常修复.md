# WebSocket线程池关闭异常修复

## 文档信息
- **文档编号**: 001
- **日期**: 2026-04-07
- **功能/模块**: WebSocket连接管理
- **完成/解决的问题**: 修复应用关闭时WebSocket回调向已终止线程池提交任务导致的RejectedExecutionException

## 问题描述

### 异常现象
应用关闭时出现大量`RejectedExecutionException`异常:

```
java.util.concurrent.RejectedExecutionException: Task ... rejected from 
java.util.concurrent.ScheduledThreadPoolExecutor@22b0fe7e[Terminated, pool size = 0, ...]
```

### 根本原因
1. Spring容器关闭时,先关闭线程池(`@PreDestroy`方法中的`shutdown()`调用)
2. WebSocket连接尚未完全关闭,仍在接收回调事件(`onOpen`, `onFailure`等)
3. WebSocket回调方法尝试向已终止的线程池提交任务:
   - `restorePublicOperations()` 使用 `websocketConnectScheduler`
   - `schedulePublicReconnect()` 使用 `reconnectScheduler`
   - `scheduleBusinessReconnect()` 使用 `reconnectScheduler`
   - `schedulePrivateReconnect()` 使用 `reconnectScheduler`
4. 线程池已终止,拒绝接收新任务,抛出异常

### 影响范围
- 应用关闭过程中产生大量异常日志
- 虽然不影响应用正常关闭,但日志污染严重
- 可能导致监控告警误报

## 解决方案

### 核心思路
采用"关闭标志 + 线程池状态检查 + 异常捕获"三重防护机制:

1. **关闭标志**: 添加`isShuttingDown`标志,在关闭开始时设置
2. **线程池状态检查**: 提交任务前检查线程池是否已关闭
3. **异常捕获**: 捕获`RejectedExecutionException`,优雅处理

### 实现细节

#### 1. 添加关闭标志
```java
// 应用关闭标志 - 用于防止在关闭过程中提交任务到已终止的线程池
private final AtomicBoolean isShuttingDown = new AtomicBoolean(false);
```

#### 2. 优化@PreDestroy方法
调整关闭顺序,确保WebSocket先关闭,再关闭线程池:

```java
@PreDestroy
public void cleanup() {
    // 1. 首先设置关闭标志,防止新的任务提交
    isShuttingDown.set(true);
    
    // 2. 关闭所有WebSocket连接,防止回调继续提交任务
    // 关闭 publicWebSocket, bussinessWebSocket, privateWebSocket
    
    // 3. 等待一小段时间,让WebSocket回调完成
    Thread.sleep(500);
    
    // 4. 关闭线程池
    // shutdown pingScheduler, reconnectScheduler, websocketConnectScheduler
}
```

#### 3. 修改任务提交方法
在所有向线程池提交任务的方法中添加检查:

**修改的方法列表**:
- `restorePublicOperations()`
- `restorePrivateOperations()`
- `restoreBusinessOperations()`
- `schedulePublicReconnect()`
- `scheduleBusinessReconnect()`
- `schedulePrivateReconnect()`

**检查模式**:
```java
private void someMethod() {
    // 检查应用是否正在关闭
    if (isShuttingDown.get()) {
        logger.info("应用正在关闭,跳过XXX操作");
        return;
    }

    // 检查线程池是否已关闭
    if (scheduler.isShutdown() || scheduler.isTerminated()) {
        logger.warn("调度器已关闭,无法执行XXX操作");
        return;
    }

    try {
        scheduler.submit(() -> {
            // 任务逻辑
        });
    } catch (RejectedExecutionException e) {
        logger.warn("无法提交XXX任务,线程池可能已关闭: {}", e.getMessage());
    } catch (Exception e) {
        logger.error("XXX操作失败", e);
    }
}
```

## 技术要点

### 1. 为什么使用AtomicBoolean
- 保证多线程环境下的可见性
- 避免使用synchronized带来的性能开销
- 简单的布尔标志,不需要复杂的同步机制

### 2. 为什么要等待500ms
- 给WebSocket回调一个完成的时间窗口
- 避免回调在线程池关闭后仍然尝试提交任务
- 500ms是一个经验值,可以根据实际情况调整

### 3. 为什么需要三重防护
- **关闭标志**: 最快的检查,避免不必要的操作
- **线程池状态检查**: 防止向已关闭的线程池提交任务
- **异常捕获**: 最后一道防线,即使前两层失败也能优雅处理

### 4. 线程池关闭顺序
```
1. isShuttingDown.set(true)  // 设置标志
2. 关闭WebSocket连接          // 停止新的回调
3. Thread.sleep(500)         // 等待回调完成
4. shutdown线程池             // 关闭线程池
5. awaitTermination          // 等待任务完成
6. shutdownNow               // 强制关闭(如果超时)
```

## 后续优化建议

### 1. 添加关闭超时配置
当前等待时间是硬编码的500ms,可以改为配置项:
```properties
websocket.shutdown.wait-time-ms=500
```

### 2. 添加关闭状态监控
记录关闭过程中的各个阶段,便于问题排查:
```java
logger.info("关闭进度: 1/5 设置关闭标志");
logger.info("关闭进度: 2/5 关闭WebSocket连接");
// ...
```

### 3. 考虑使用CountDownLatch
如果需要更精确地等待WebSocket回调完成,可以使用CountDownLatch:
```java
private final CountDownLatch shutdownLatch = new CountDownLatch(3); // 3个WebSocket

// 在WebSocket关闭回调中
shutdownLatch.countDown();

// 在cleanup方法中
shutdownLatch.await(5, TimeUnit.SECONDS);
```

## 总结
通过添加关闭标志、优化关闭顺序、添加线程池状态检查和异常捕获,成功解决了应用关闭时的RejectedExecutionException问题。修改后的代码更加健壮,能够优雅地处理应用关闭场景。
